Documento de Requisitos: Motor Preditivo de Futebol (Arquitetura Bayesiana de Alta Precisão)1. Visão Geral do ProdutoO sistema é um motor preditivo focado em precisão absoluta para simulação e ranqueamento de seleções de futebol. Ao remover as restrições de tempo de resposta em tempo real, o modelo utiliza Inferência Bayesiana com cadeias de Markov (MCMC) para calcular distribuições de probabilidade densas sobre a força de ataque e defesa de cada equipe. As simulações de confrontos são feitas através de processamento assíncrono massivo e paralelo, garantindo a captura precisa de anomalias estatísticas ("zebras").2. Arquitetura TecnológicaA stack foi desenhada para separar a matemática profunda da simulação de força bruta, rodando em um ambiente conteinerizado e otimizado para ambientes Linux (como Pop!_OS ou Parrot OS, garantindo estabilidade no gerenciamento de processos).Core Estatístico (Treinamento MCMC): Python, utilizando a biblioteca PyMC ou Stan. Focado em rodar em background para atualizar as distribuições de probabilidade (matemática pesada).Motor de Simulação & Backend: Go. Escolhido por sua concorrência nativa (Goroutines), capaz de rodar milhões de simulações de Monte Carlo explorando 100% dos múltiplos núcleos da CPU sem gargalos.Mensageria e Filas: RabbitMQ ou Redis, operando o controle de jobs assíncronos entre a API, o motor Go e os workers Python.Banco de Dados: PostgreSQL (armazenamento persistente de placares, variáveis socioculturais e parâmetros calculados).3. Especificações da Modelagem MatemáticaO agente deve implementar um modelo hierárquico Bayesiano baseado na estrutura de Dixon-Coles.3.1 A Taxa de Gols (Poisson)O número de gols marcados pelo time $i$ contra o time $j$ segue uma distribuição de Poisson, onde a taxa esperada $\lambda$ é calculada no log-espaço para garantir valores positivos:$$\log(\lambda_{ij}) = \text{baseline} + \alpha_i - \beta_j + \gamma$$Onde $\alpha_i$ é o ataque de $i$, $\beta_j$ é a defesa de $j$, e $\gamma$ é o fator mando de campo/continente.3.2 Modelagem Hierárquica e Contexto SocioculturalPara incluir os fatores de "camisa" e socioeconômicos sem depender de confrontos diretos prévios, os parâmetros não são estáticos, mas extraídos de uma distribuição normal parametrizada pelas características contextuais (Priors informativas):$$\alpha_i \sim \mathcal{N}\left(\mu_{ataque} + \sum_{k=1}^{n} w_k x_{ik}, \sigma^2_\alpha\right)$$Nesta fórmula, $x_{ik}$ representa o vetor de características socioculturais do time (ex: histórico de vitórias, PIB per capita, qualidade da liga), e $w_k$ é o peso que o modelo aprenderá para cada variável.4. Requisitos Funcionais (RF)RF01 - Ingestão de Dados de Partida: Uma rota em Go que recebe resultados reais de jogos e os salva no PostgreSQL, sinalizando a fila de processamento que novos dados estão disponíveis.RF02 - Treinamento MCMC em Lote (Assíncrono): Um worker em Python que consome a base de jogos e recalibra as distribuições de probabilidade de todos os parâmetros ($\alpha, \beta, \gamma, w$). Este processo deve rodar em janelas programadas (ex: fim do dia) e gerar distribuições posteriores (posterior distributions), não apenas um número fixo.RF03 - Atualização de Ranking Global: Após o fim do job MCMC, o sistema deve calcular a média da distribuição posterior do ataque menos a média da defesa para ordenar o ranking global.RF04 - Simulação Massiva Paralelizada: Ao solicitar um confronto direto, o motor em Go deve lançar Goroutines distribuídas por todos os núcleos disponíveis da CPU para amostrar aleatoriamente as distribuições e simular a partida de 1.000.000 a 5.000.000 de vezes.RF05 - Output via Polling/Webhook: A requisição de simulação deve devolver imediatamente um job_id. O cliente fará o polling deste ID até que o motor em Go finalize a simulação massiva e retorne o JSON com as probabilidades exatas (vitórias, empate, placares prováveis).5. Requisitos Não Funcionais (RNF)RNF01 - Maximização de Multi-Core: O código Go deve utilizar a diretiva runtime.GOMAXPROCS para alocar e saturar todos os núcleos lógicos do host durante a simulação de Monte Carlo. O PyMC deve configurar o parâmetro chains para rodar simultaneamente de acordo com o número de núcleos.RNF02 - Arquitetura Desacoplada e Tolerante a Falhas: A queda de um worker Python durante uma inferência de 2 horas não deve derrubar o sistema. O RabbitMQ deve reencaminhar o job caso não receba um ack de sucesso.RNF03 - Imutabilidade Histórica: Os dados brutos inseridos no banco nunca devem ser alterados pelas inferências do modelo. O sistema deve manter o histórico intacto para permitir o re-treinamento total do zero caso a modelagem seja refinada no futuro.
